# 需求文档：服务器启动预检

## 简介

服务器启动时执行全面的依赖预检，确保所有外部依赖（PostgreSQL、MinIO、环境变量、数据库迁移）就绪后才开始监听 HTTP 请求。任何预检失败立即报错退出，避免"启动成功但运行时才发现依赖不可用"的问题。

## 需求

### 需求 1 - PostgreSQL 连通性检查

**用户故事：** 作为运维人员，我想要服务器启动时自动检查数据库连接，以便在数据库不可用时立即发现而非等到用户请求失败。

#### 验收标准

1. 当服务器启动时，系统应该尝试连接 PostgreSQL 并执行一条简单查询（如 `SELECT 1`）
2. 如果数据库连接失败，系统应该输出明确的错误日志（如 `❌ PostgreSQL 连接失败: <具体错误>`）并以 exit code 1 退出
3. 如果数据库连接成功，系统应该输出 `✅ PostgreSQL 连接正常`

### 需求 2 - MinIO/S3 连通性检查与 Bucket 初始化

**用户故事：** 作为运维人员，我想要服务器启动时自动检查 MinIO 可用性并确保 bucket 存在，以便文件上传功能在首次使用前就已就绪。

#### 验收标准

1. 当服务器启动时，系统应该尝试连接 MinIO 并执行 `HeadBucket` 检查连通性
2. 如果 MinIO 连接失败，系统应该输出明确的错误日志（如 `❌ MinIO 连接失败: <具体错误>`）并以 exit code 1 退出
3. 如果 bucket 不存在，系统应该自动创建 bucket
4. 如果 MinIO 连接成功且 bucket 就绪，系统应该输出 `✅ MinIO 连接正常，bucket "pet-uploads" 已就绪`

### 需求 3 - 关键环境变量检查

**用户故事：** 作为运维人员，我想要服务器启动时检查必要的环境变量是否已配置，以便在部署配置不完整时立即发现。

#### 验收标准

1. 当服务器在生产环境（`NODE_ENV=production`）启动时，系统应该检查以下环境变量已设置：`DATABASE_URL`、`JWT_SECRET`、`WX_APPID`、`WX_SECRET`、`S3_ENDPOINT`、`S3_ACCESS_KEY`、`S3_SECRET_KEY`
2. 如果有缺失的环境变量，系统应该输出所有缺失项的列表（如 `❌ 缺少环境变量: WX_APPID, WX_SECRET`）并以 exit code 1 退出
3. 在开发环境下（非 production），环境变量检查应该跳过（因为有默认值兜底）

### 需求 4 - 数据库迁移状态检查

**用户故事：** 作为运维人员，我想要服务器启动时检查数据库迁移是否已应用到最新版本，以便在迁移遗漏时及时发现。

#### 验收标准

1. 当服务器启动时，系统应该检查 Drizzle ORM 的迁移状态，确认所有迁移文件都已应用
2. 如果有未应用的迁移，系统应该输出警告日志（如 `⚠️ 有 N 个未应用的数据库迁移`）但不阻止启动（仅警告，因为迁移需要通过 `pnpm db:migrate` 手动执行）

### 需求 5 - 预检执行时序

**用户故事：** 作为开发者，我想要预检在 HTTP 服务启动前完成，以便确保服务开始接受请求时所有依赖都已就绪。

#### 验收标准

1. 当服务器启动时，系统应该先执行所有预检，全部通过后才开始监听端口
2. 预检的执行顺序为：环境变量检查 → PostgreSQL 检查 → MinIO 检查 → 数据库迁移检查
3. 如果任一阻断性预检失败（环境变量、PG、MinIO），系统应该立即退出，不执行后续检查
4. 预检开始时输出 `🔍 正在检查服务依赖...`，全部通过后输出 `✅ 所有依赖检查通过`
