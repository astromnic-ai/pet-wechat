# 需求文档 — 管理后台

## 简介

为 YEHEY 宠物"在场"管理后台补全运营管理功能，覆盖系统概览、行为日程管理、图像审核中心、定制中心、设备管理、用户管理和数据看板。沿用现有 Admin Key 认证、React + Ant Design 技术栈。

---

## P0 需求

### 需求 1 — 系统概览（增强版）

**用户故事：** 作为管理员，我想在首页看到系统核心 KPI 和基础运营数据，以便快速了解当前系统状态。

#### 验收标准

1. 当管理员进入系统概览页时，系统应该展示以下 KPI 卡片：注册用户总数、已绑定设备用户数、已定制用户数（petAvatars status=done 的 distinct userId）、宠物总数
2. 当管理员进入系统概览页时，系统应该展示：近一周活跃设备数（最近 7 天有行为记录的设备）、今日互动总数（今日 petBehaviors 条数）
3. 当管理员进入系统概览页时，系统应该展示设备分布卡片：桌面端 vs 项圈的数量与占比（饼图或统计数字）
4. 当管理员进入系统概览页时，系统应该展示设备活跃度分布：基于最近 3 天有行为记录 → 高活跃，3-7 天 → 中活跃，>7 天或无记录 → 低活跃

### 需求 2 — 行为日程管理

**用户故事：** 作为管理员，我想按宠物类型（猫/狗/其他）配置每日行为时间轴，以便桌面设备按日程播放对应动画。

#### 验收标准

1. 当管理员进入行为日程页时，系统应该展示按宠物类型（猫/狗/其他）分组的日程列表
2. 当管理员创建/编辑日程时，系统应该提供时间轴编辑器，支持配置 24 小时内每个时间段对应的动作类型
3. 系统应该支持 14 种动作类型：蹲坐（sit）、吃饭（eat）、睡觉（sleep）、趴卧（lie）、跑（run）、走（walk）、玩球（play_ball）、噗噗（poop）、看电视（watch_tv）、追尾巴（chase_tail）、挠空气（scratch_air）、做美梦（dream）、舔爪子（lick_paw）、转圈（spin）
4. 当管理员编辑时间轴时，系统应该支持拖拽添加时间块、拉伸/缩小时间块、删除时间块
5. 当管理员设置日程时，系统应该支持选择生效策略：每天、仅工作日、仅节假日
6. 当管理员保存日程后，系统应该支持"应用当前配置"操作，将该日程标记为当前生效日程
7. 设备通过 `GET /api/schedules/current?species=cat` 轮询获取当前生效日程，后端返回对应宠物类型的活跃日程配置

### 需求 3 — 图像审核中心

**用户故事：** 作为管理员，我想审核用户上传的宠物定制图片，通过或拒绝并告知原因，以便控制内容质量。

#### 验收标准

1. 当管理员进入图像审核页时，系统应该在顶部展示：今日待处理数、今日已完成数、已同步设备总数（status=done 的 petAvatars 数量）
2. 当管理员进入图像审核页时，系统应该按分类展示：待审核（pending）、已通过（approved）、已拒绝（rejected）标签页
3. 每个审核项应该展示：图片缩略图（sourceImageUrl）、宠物信息（名称、类型）、上传时间、审核状态
4. 当管理员点击"通过"时，系统应该将 petAvatar status 更新为 approved，并创建一条系统消息通知用户
5. 当管理员点击"拒绝"时，系统应该弹出拒绝原因选择框（预设原因 + 手动填写），将 status 更新为 rejected，并创建通知消息附带拒绝原因
6. 当管理员点击"下载"时，系统应该支持下载原图到本地

> 注意：petAvatars 表当前 status 枚举为 pending|processing|done|failed，需要新增 approved 和 rejected 状态。审核通过（approved）后进入定制池，定制完成后变为 done。

### 需求 4 — 定制中心

**用户故事：** 作为设计师/运营，我想管理宠物的动作定制任务，为每个宠物上传各动作素材，完成后一键同步给用户。

#### 验收标准

1. 当管理员进入定制中心时，系统应该展示：累计定制总量、今日新增待定制数量
2. 系统应该以卡片列表展示定制任务，每个卡片包含：上传原图、用户信息、宠物信息、定制分类、任务状态（待处理/进行中/已完成）
3. 系统应该支持按定制类型筛选任务
4. 当管理员选中一个任务后，右侧工作区应该展示该宠物的动作定制进度：14 种动作的完成状态
5. 管理员可以为每个动作上传素材图片（对应 petAvatarActions），上传后该动作标记为已完成
6. 当所有动作定制完成后，管理员可以点击"一键同步"，系统应该将 petAvatar status 更新为 done，并通过消息系统通知用户
7. 任务状态流转：pending（待审核）→ approved（审核通过，待定制）→ processing（定制中，有部分动作已上传）→ done（全部完成并同步）

### 需求 5 — 设备管理（增强版）

**用户故事：** 作为管理员，我想通过多维筛选快速检索设备，查看详情并管理设备。

#### 验收标准

1. 当管理员进入设备管理页时，系统应该提供组合筛选：设备型号（桌面宠物/项圈）、设备绑定状态（已绑定/未绑定）、宠物类型（猫/狗/其他）、设备状态（在线/离线）、注册时间排序
2. 设备列表应该展示：设备名称、MAC 地址、型号、绑定状态、在线状态、绑定用户、绑定宠物、最后在线时间
3. 当管理员点击设备行时，应该展示设备详情侧边栏/弹窗，包含完整设备信息
4. 系统应该支持在列表中实时刷新设备状态

## P1 需求（精简版）

### 需求 6 — 系统概览实时动态

**用户故事：** 作为管理员，我想在概览页看到最近的运营动态，以便掌握系统变化趋势。

#### 验收标准

1. 当管理员查看概览页时，系统应该展示实时动态列表，包含：新设备上线、新增图像审核、最近行为事件
2. 动态列表按时间倒序展示最近 20 条记录
3. 系统应该支持手动刷新（P1 不要求自动轮询）

### 需求 7 — 数据看板

**用户故事：** 作为管理员，我想查看核心运营指标的可视化图表，以便做数据驱动的决策。

#### 验收标准

1. 当管理员进入数据看板页时，系统应该展示：实时在线设备数（1 小时内有行为记录的设备）
2. 系统应该展示：平均互动数值 = 今日 behavior 总数 / 今日在线用户数
3. 系统应该展示：一周内互动数排行榜（按用户或设备维度，取 Top 10）
4. 系统应该以图表形式（柱状图/折线图）展示近 7 天每日互动趋势

### 需求 8 — 用户管理（增强版）

**用户故事：** 作为管理员，我想查看用户详细信息和关联的宠物与设备。

#### 验收标准

1. 用户列表应该展示：微信信息（头像、昵称）、注册时间、账号状态、绑定设备数、宠物数
2. 当管理员点击用户行时，应该展示用户详情，包含其关联的宠物列表和设备列表

## P2 需求（占位）

### 需求 9 — 系统健康状态

**用户故事：** 作为管理员，我想在概览页看到系统组件健康状况。

#### 验收标准

1. 系统应该在概览页展示健康状态面板（占位 UI，显示"功能开发中"）
2. 预留：服务器状态、API 响应时间、数据库连接、云存储空间

### 需求 10 — 会员权益配置

**用户故事：** 作为管理员，我想配置用户的会员等级和权益。

#### 验收标准

1. 用户详情页应该展示会员信息区域（占位 UI，显示"功能开发中"）
2. 预留：会员等级、到期时间、绑定设备上限、定制次数配置
